27: [Heading 2] 1. Tính cấp thiết của đề tài
28: [Normal] Trong bối cảnh chuyển đổi số diễn ra mạnh mẽ trên toàn cầu, các ứng dụng web đã trở thành xương sống của hầu hết các hoạt động kinh doanh, thương mại điện tử, ngân hàng trực tuyến, dịch vụ công và truyền thông xã hội. Theo thống kê từ Microsoft và các tổ chức an ninh mạng uy tín, mỗi ngày có hàng trăm nghìn cuộc tấn công nhắm vào các ứng dụng web, với mức độ tinh vi ngày càng gia tăng. Các lỗ hổng bảo mật như thiếu hụt HTTP Security Headers (HSTS, CSP, X-Frame-Options), SQL Injection, Cross-Site Scripting (XSS), Cross-Site Request Forgery (CSRF) và lỗi phân quyền (Broken Access Control) vẫn liên tục xuất hiện trong các báo cáo OWASP Top 10 qua nhiều năm, cho thấy đây là những vấn đề kinh niên chưa có giải pháp triệt để.
29: [Normal] Việc kiểm thử bảo mật thủ công (manual penetration testing) đòi hỏi kiến thức chuyên môn sâu, thời gian dài và chi phí cao, khó có thể triển khai thường xuyên trong quy trình phát triển phần mềm hiện đại (DevOps/DevSecOps). Các công cụ kiểm thử tự động hiện có trên thị trường thường là giải pháp thương mại đắt tiền hoặc mã nguồn đóng, khó tùy chỉnh cho nhu cầu cụ thể. Xuất phát từ thực tế đó, đề tài này hướng tới xây dựng một hệ thống kiểm thử bảo mật web dòng lệnh (CLI) mã nguồn mở với kiến trúc module hóa linh hoạt, đồng thời ứng dụng mô hình Behavior Tree (Cây hành vi) - một phương pháp sinh và quản lý test case có tính biểu đạt cao, có nguồn gốc từ robotics và trí tuệ nhân tạo trong game - để nâng cao khả năng tự động hóa và mở rộng của hệ thống.
30: [Normal] 2. Mục tiêu nghiên cứu
31: [Heading 2] Đề tài được thực hiện với các mục tiêu cụ thể sau đây, được phân chia thành hai nhóm chính: lý thuyết và thực nghiệm.
32: [Normal] - Về mặt lý thuyết: (a) Nghiên cứu tổng quan về kiến trúc bảo mật ứng dụng web, các lớp lỗ hổng phổ biến theo phân loại OWASP Top 10 và MITRE ATT&CK, bao gồm cơ chế hoạt động, điều kiện khai thác và phương pháp phòng ngừa cho từng lớp lỗ hổng. (b) Nghiên cứu chuyên sâu về mô hình Behavior Tree (Cây hành vi) - lịch sử phát triển, cấu trúc đồ thị có hướng, các loại node điều khiển (Sequence, Selector, Parallel, Decorator), cơ chế truyền trạng thái (Success/Failure/Running) và các ứng dụng thực tế trong robotics, game AI và kiểm thử phần mềm. (c) Nghiên cứu các mẫu kiến trúc trong Python để xây dựng CLI framework có khả năng mở rộng (plugin-based architecture).
33: [Normal] - Về mặt thực nghiệm: (a) Thiết kế và xây dựng thành công công cụ WebSec Test CLI với 5 module kiểm thử chính bao gồm Headers (kiểm tra HTTP security headers), Auth (phát hiện lỗ hổng xác thực), CSRF (kiểm tra Cross-Site Request Forgery), Injection (SQLi, XSS, Command Injection) và Authz (kiểm tra phân quyền). (b) Tích hợp Behavior Tree Engine cho phép định nghĩa luồng test case dưới dạng cây đồ thị, hỗ trợ thực thi tuần tự (Sequence), thực thi fallback (Selector) và thực thi đồng thời (Parallel). (c) Xây dựng bộ 184 test cases (pytest) bao gồm unit test, integration test và full-stack test, đảm bảo chất lượng mã nguồn với tỷ lệ pass 100%. (d) Tạo báo cáo kết quả dạng color-coded terminal và JSON chi tiết phục vụ CI/CD integration.
34: [Normal] 3. Đối tượng và phạm vi nghiên cứu
35: [Heading 2] Đối tượng nghiên cứu của đề tài bao gồm: (1) Các kỹ thuật kiểm thử bảo mật ứng dụng web phía máy chủ (server-side security testing), tập trung vào 5 nhóm lỗ hổng chính: thiếu HTTP Security Headers, lỗ hổng xác thực (Authentication), Cross-Site Request Forgery (CSRF), Injection (SQL, XSS, Command) và lỗ hổng phân quyền (Authorization). (2) Mô hình Behavior Tree và các biến thể của nó trong việc biểu diễn luồng kiểm thử. (3) Kiến trúc plugin-based trong Python sử dụng thư viện requests cho HTTP client.
36: [Normal] Phạm vi nghiên cứu được giới hạn như sau: (1) Hệ thống được xây dựng bằng ngôn ngữ Python phiên bản 3.10+, tương thích với Windows và Linux. (2) Các module kiểm thử hoạt động ở chế độ non-destructive (chỉ đọc - read-only), chỉ gửi HTTP requests và phân tích responses, không thực hiện ghi, sửa, xóa dữ liệu hay thay đổi trạng thái ứng dụng đích. (3) Behavior Tree Engine hỗ trợ 4 loại node cơ bản: ActionNode (node lá thực thi test case), SequenceNode (thực thi tuần tự theo thứ tự), SelectorNode (thực thi với cơ chế fallback), ParallelNode (thực thi đồng thời đa luồng). (4) Hệ thống chưa hỗ trợ kiểm thử API (REST, GraphQL), WebSocket, authentication phức tạp (OAuth 2.0, JWT) hay giao diện người dùng đồ họa (GUI).
37: [Heading 2] 4. Phương pháp nghiên cứu
38: [Normal] Đề tài sử dụng kết hợp các phương pháp nghiên cứu lý thuyết và thực nghiệm:
39: [Normal] - Phương pháp nghiên cứu lý thuyết: Thu thập, phân tích và tổng hợp tài liệu kỹ thuật từ các nguồn uy tín quốc tế bao gồm OWASP (Open Web Application Security Project) với các ấn phẩm OWASP Top 10 và OWASP Testing Guide; MITRE ATT&CK Framework; tài liệu Behavior Tree từ các công trình nghiên cứu của Champandard (2007) và Colledanchise & Oegren (2018); tài liệu MSDN của Microsoft về API CreateProcess và Windows Service Architecture; cùng các kho mã nguồn mở tham khảo.
40: [Normal] - Phương pháp thực nghiệm: (a) Sử dụng Python với thư viện requests để xây dựng HTTP client có hỗ trợ session management, cookie handling và CSRF token extraction. (b) Sử dụng thư viện responses để mock HTTP trong unit test, đảm bảo các test case chạy độc lập không phụ thuộc mạng. (c) Áp dụng mô hình Behavior Tree vào thiết kế luồng kiểm thử với các node tương ứng cho từng bước kiểm tra. (d) Xây dựng Flask application giả lập có chủ đích cài đặt các lỗ hổng bảo mật để kiểm thử tích hợp (integration testing). (e) Sử dụng pytest framework để tự động hóa kiểm thử với bộ 184 test cases. (f) Áp dụng kiến trúc module (plugin-based) để tổ chức mã nguồn, cho phép dễ dàng thêm module kiểm thử mới mà không ảnh hưởng đến module hiện có.

57: [Heading 1] CHƯƠNG 2: CƠ SỞ LÝ THUYẾT VỀ BẢO MẬT WEB VÀ BEHAVIOR TREE
58: [Heading 2] 2.1. Tổng quan về bảo mật ứng dụng Web
59: [Normal] Bảo mật ứng dụng web (Web Application Security) là một lĩnh vực quan trọng trong an toàn thông tin, tập trung vào việc bảo vệ các ứng dụng web khỏi các mối đe dọa và tấn công từ Internet. Khác với bảo mật hạ tầng mạng (network security) tập trung vào tường lửa và IDS/IPS, bảo mật ứng dụng web xử lý các lỗ hổng ở tầng ứng dụng (application layer - Layer 7 của mô hình OSI) - nơi mà các giải pháp bảo mật truyền thống thường không phát hiện được.
60: [Normal] Theo OWASP Top 10 - 2021, 10 nhóm lỗ hổng ứng dụng web phổ biến nhất bao gồm: (1) Broken Access Control - lỗi phân quyền; (2) Cryptographic Failures - lỗi mã hóa; (3) Injection - tiêm nhiễm mã độc (SQL, NoSQL, OS Command); (4) Insecure Design - thiết kế không an toàn; (5) Security Misconfiguration - cấu hình bảo mật sai; (6) Vulnerable and Outdated Components - thành phần lỗi thời; (7) Identification and Authentication Failures - lỗi xác thực; (8) Software and Data Integrity Failures - lỗi toàn vẹn; (9) Security Logging and Monitoring Failures - thiếu giám sát; (10) Server-Side Request Forgery (SSRF). Các công cụ kiểm thử tự động (DAST - Dynamic Application Security Testing) đóng vai trò quan trọng trong việc phát hiện sớm các lỗ hổng này, đặc biệt là trong quy trình DevSecOps khi ứng dụng được triển khai liên tục (continuous deployment).
61: [Heading 3] 2.1.1. Lỗ hổng thiếu Header bảo mật (Security Headers)
62: [Normal] HTTP Security Headers là các trường đặc biệt trong HTTP response mà máy chủ web gửi kèm để hướng dẫn trình duyệt thực thi các chính sách bảo mật nhất định. Đây là lớp phòng thủ quan trọng ở phía client, giúp giảm thiểu nhiều dạng tấn công phổ biến. Các header bảo mật quan trọng nhất bao gồm:
63: [Normal] - Strict-Transport-Security (HSTS): Yêu cầu trình duyệt chỉ kết nối qua HTTPS, ngăn chặn tấn công SSL Strip và Man-in-the-Middle (MITM). Ví dụ cấu hình: Strict-Transport-Security: max-age=31536000; includeSubDomains. Module Header trong WebSec Test kiểm tra sự hiện diện, giá trị max-age (khuyến nghị >= 1 nam) và thuộc tính includeSubDomains.
64: [Normal] - Content-Security-Policy (CSP): Cho phép quản trị viên chỉ định chính xác nguồn tài nguyên nào (script, style, image, font, frame) được phép tải và thực thi. CSP là phòng thủ mạnh nhất chống XSS (Cross-Site Scripting). Module kiểm tra sự hiện diện của CSP header và đánh giá mức độ chi tiết của policy.
65: [Normal] - X-Frame-Options: Ngăn chặn tấn công Clickjacking bằng cách kiểm soát trang web có được hiển thị trong frame/iframe hay không. Các giá trị: DENY (từ chối toàn bộ), SAMEORIGIN (chỉ cho phép cùng nguồn gốc).
66: [Normal] - X-Content-Type-Options: Ngăn chặn trình duyệt thực hiện MIME sniffing - hành vi tự động phát hiện kiểu nội dung, có thể bị lợi dụng để thực thi mã độc. Giá trị duy nhất là nosniff.
67: [Normal] - Referrer-Policy: Kiểm soát thông tin Referrer được gửi kèm khi người dùng điều hướng giữa các trang, giúp bảo vệ thông tin nhạy cảm trên URL khỏi bị rò rỉ.
68: [Normal] Module Headers của WebSec Test tự động kiểm tra đồng thời 5 header bảo mật này trên tất cả các endpoint được phát hiện. Mỗi header là một ActionNode độc lập trong Behavior Tree, cho phép kiểm tra song song (thông qua ParallelNode) và ghi nhận kết quả riêng biệt. Kết quả được đánh giá dựa trên cấu hình khuyến nghị của OWASP và Mozilla Observatory.
69: [Heading 3] 2.1.2. Lỗ hổng xác thực (Authentication)
70: [Normal] Xác thực (Authentication) là quá trình xác minh danh tính của người dùng hoặc hệ thống. Đây là lớp bảo vệ đầu tiên và quan trọng nhất của mọi ứng dụng web. Các lỗ hổng xác thực phổ biến bao gồm:
71: [Normal] - Form login gửi thông tin qua HTTP không mã hóa: Mật khẩu và tên đăng nhập được gửi dưới dạng plaintext, có thể bị đánh cắp thông qua tấn công MITM. Module Auth kiểm tra xem form login có thuộc tính method=POST và action sử dụng HTTPS không.
72: [Normal] - Chấp nhận mật khẩu trống (blank password): Nhiều ứng dụng trong quá trình phát triển cho phép tài khoản không có mật khẩu, sau đó quên không kiểm tra lại khi triển khai production. Module Auth thử đăng nhập với username phổ biến (admin, user, test) và password trống.
73: [Normal] - SQL Injection trong truy vấn xác thực: Khi tham số username/password được nối trực tiếp vào câu truy vấn SQL mà không qua kiểm tra, kẻ tấn công có thể sử dụng các payloads như ' OR 1=1 -- để bypass xác thực. Module Auth thử nghiệm 10+ payloads SQLi phổ biến cho mục đích bypass login.
74: [Normal] - Brute Force không giới hạn: Hệ thống không có cơ chế khóa tài khoản hoặc rate-limiting sau số lần đăng nhập thất bại. Module Auth hiện phát hiện form login, trong tương lai có thể mở rộng kiểm tra rate-limiting bằng cách gửi nhiều request liên tiếp.
75: [Normal] Các bước kiểm thử trong module Auth được tổ chức thành SequenceNode vì chúng phụ thuộc nhau: phát hiện form login -> kiểm tra blank password -> thử SQLi bypass. Nếu bước phát hiện form login thất bại, các bước sau không cần thực thi.
76: [Heading 3] 2.1.3. Lỗ hổng CSRF (Cross-Site Request Forgery)
77: [Normal] CSRF (Cross-Site Request Forgery) là một kỹ thuật tấn công trong đó kẻ tấn công lừa nạn nhân đã xác thực thực hiện các hành động không mong muốn trên ứng dụng web mục tiêu. Cụ thể, kẻ tấn công tạo một trang web độc hại chứa các form ẩn hoặc request tự động gửi đến ứng dụng mục tiêu. Vì nạn nhân đã có session hợp lệ (thông qua cookie), máy chủ xem request này như một request hợp lệ từ chính nạn nhân.
78: [Normal] Biện pháp phòng thủ chính chống CSRF là sử dụng CSRF token - một giá trị ngẫu nhiên, duy nhất cho mỗi session hoặc mỗi request, được nhúng trong form HTML dưới dạng hidden input field. Khi form được submit, máy chủ kiểm tra token gửi lên có khớp với token đã tạo không. Ngoài ra, các biện pháp bổ sung bao gồm: sử dụng SameSite cookie attribute (Strict/Lax), kiểm tra Referer/Origin header, và sử dụng CAPTCHA cho các hành động nhạy cảm.
79: [Normal] Module CSRF trong WebSec Test thực hiện hai kiểm tra chính: (1) Phát hiện các form POST không có CSRF token - mỗi form được phân tích HTML để tìm hidden input field chứa token (thường có tên chứa 'csrf', 'token', 'authenticity_token', '_token'). (2) Kiểm tra khả năng tái sử dụng token - lấy token từ một form, sau đó sử dụng lại token đó trong request thứ hai để kiểm tra xem máy chủ có chấp nhận token cũ không. Cả hai kiểm tra được tổ chức trong SelectorNode, cho phép fail độc lập.
80: [Heading 3] 2.1.4. Lỗ hổng Injection (SQL, XSS, Command)
81: [Normal] Injection là nhóm lỗ hổng nguy hiểm nhất trong OWASP Top 10, xảy ra khi dữ liệu không được kiểm tra hoặc làm sạch được gửi trực tiếp vào interpreter (trình thông dịch). WebSec Test kiểm tra ba dạng injection chính:
82: [Normal] - SQL Injection: Xảy ra khi dữ liệu đầu vào được nối trực tiếp vào câu truy vấn SQL. Kẻ tấn công có thể chèn các mệnh đề SQL để: bypass xác thực (' OR 1=1 --), truy vấn dữ liệu trái phép (UNION SELECT), hoặc thao túng cơ sở dữ liệu (DROP TABLE, UPDATE). Module Injection sử dụng 15+ payloads SQLi phổ biến, gửi qua tham số GET/POST và phân tích HTTP response để tìm dấu hiệu lỗi SQL (SQL syntax error, database error messages).
83: [Normal] - Cross-Site Scripting (XSS): Kẻ tấn công chèn mã JavaScript độc hại vào trang web, có thể đánh cắp cookie, chuyển hướng trình duyệt, hoặc thực thi hành động thay mặt nạn nhân. Có 3 loại XSS: Reflected (phản hồi ngay lập tức), Stored (lưu trên máy chủ), DOM-based (phía client). Module Injection kiểm tra Reflected XSS bằng cách gửi payloads chứa thẻ script (<script>alert(1)</script>) và event handler (onerror, onload) qua form inputs, sau đó kiểm tra response có chứa payload không được mã hóa (unescaped).
84: [Normal] - Command Injection: Kẻ tấn công chèn lệnh hệ thống vào input của ứng dụng. Khi ứng dụng gọi hàm system() hoặc subprocess mà không kiểm tra đầu vào, kẻ tấn công có thể thực thi lệnh trên máy chủ. Payloads phổ biến: ; ls, | dir, && whoami, `id`. Module Injection gửi các payloads này qua form inputs và kiểm tra trong response có dấu hiệu của việc thực thi lệnh (output của lệnh hệ thống, thông tin hệ thống bị lộ).
85: [Normal] Module Injection sử dụng SelectorNode vì cả ba dạng injection đều độc lập - chỉ cần phát hiện một lỗ hổng là test case coi như có kết quả (cảnh báo). Mỗi dạng injection được thực thi như một ActionNode riêng với bộ payload riêng.
86: [Heading 3] 2.1.5. Lỗ hổng phân quyền (Authorization)
87: [Normal] Authorization (Phân quyền) là quá trình xác định người dùng đã xác thực được phép thực hiện hành động gì và truy cập tài nguyên nào. Khác với Authentication (xác thực - bạn là ai), Authorization trả lời câu hỏi 'bạn được phép làm gì'. Lỗ hổng phân quyền là nhóm nguy hiểm nhất trong OWASP Top 10 - 2021 (Broken Access Control).
88: [Normal] WebSec Test kiểm tra hai dạng lỗ hổng phân quyền phổ biến:
89: [Normal] - Forced Browsing (còn gọi là Direct Object Reference): Kẻ tấn công truy cập trực tiếp vào các URL nhạy cảm (admin panel, config page, backup files) mà không cần xác thực hoặc phân quyền. Module Authz duy trì một danh sách các đường dẫn nhạy cảm phổ biến (COMMON_PATHS) bao gồm /admin, /config, /backup, /.git, /WEB-INF, /manager, /console, /api-docs, /swagger, /.env. Với mỗi đường dẫn, module gửi GET request và kiểm tra nếu response trả về HTTP 200 OK (thay vì 401 Unauthorized hoặc 403 Forbidden) thì báo lỗi.
90: [Normal] - IDOR (Insecure Direct Object Reference): Kẻ tấn công thay đổi tham số định danh trong URL (ví dụ: /user/1 -> /user/2) để truy cập dữ liệu của người dùng khác. Module Authz kiểm tra IDOR bằng cách gửi request với các ID tuần tự (1, 2, 3) và so sánh response - nếu response chứa dữ liệu người dùng khác nhau nhưng đều trả về 200 thì báo lỗi. Cả hai kiểm tra được thực thi song song thông qua ParallelNode vì chúng độc lập với nhau.
91: [Heading 2] 2.2. Behavior Tree - Cây hành vi
92: [Normal] Behavior Tree (Cây hành vi) là một mô hình toán học để biểu diễn và điều khiển luồng thực thi trong các hệ thống tự động. Được giới thiệu lần đầu trong game Halo 2 (2004) bởi Damian Isla, Behavior Tree nhanh chóng trở thành tiêu chuẩn trong ngành công nghiệp game (Spore, Halo 3, BioShock, Left 4 Dead) nhờ tính trực quan, khả năng tái sử dụng và dễ dàng mở rộng. Sau đó, Behavior Tree được ứng dụng rộng rãi trong robotics (Colledanchise & Oegren, 2018) và gần đây là trong kiểm thử phần mềm tự động.
93: [Normal] Behavior Tree là một đồ thị có hướng dạng cây (directed acyclic graph), trong đó:
94: [Normal] - Node gốc (Root): Điểm bắt đầu của cây, thực thi node con và trả về kết quả.
95: [Normal] - Node lá (Leaf): Là các hành động cụ thể (Action Node) hoặc điều kiện kiểm tra (Condition Node). Các node lá không có node con.
96: [Normal] - Node nhánh (Composite): Điều khiển luồng thực thi giữa các node con. Gồm Sequence, Selector, Parallel, Decorator.
97: [Normal] - Cơ chế truyền trạng thái: Mỗi node sau khi thực thi trả về một trong ba trạng thái: Success (thành công), Failure (thất bại), hoặc Running (đang chạy - chưa hoàn thành).
98: [Normal] Ưu điểm của Behavior Tree so với Finite State Machine (FSM) truyền thống: (1) Dễ dàng mở rộng - thêm node mới mà không cần sửa đổi node hiện tại; (2) Tái sử dụng cao - cây con có thể được dùng lại trong nhiều ngữ cảnh; (3) Trực quan - cấu trúc cây dễ hiểu và dễ debug; (4) Xử lý song song - Parallel node cho phép thực thi đồng thời; (5) Tính module - mỗi node độc lập, có thể kiểm thử riêng.
99: [Heading 3] 2.2.1. Các loại node trong Behavior Tree
100: [Normal] WebSec Test Behavior Tree Engine hỗ trợ 4 loại node cơ bản, được thiết kế để đáp ứng các nhu cầu khác nhau trong kiểm thử bảo mật:
101: [Normal] - ActionNode (Leaf Node): Là node lá thực thi một hành động cụ thể và trả về kết quả (Success/Failure/Running). Trong WebSec Test, mỗi ActionNode tương ứng với một test case đơn lẻ. Ví dụ: check_hsts_header là ActionNode kiểm tra header HSTS; test_sqli_payload là ActionNode gửi SQLi payload. ActionNode lưu trữ kết quả dưới dạng TestResult với đầy đủ thông tin: module, test_name, status, severity, endpoint, evidence, recommendation, và thời gian thực thi (execution_time_ms). Khi ActionNode gặp exception (ví dụ: network timeout), nó ghi nhận status = ERROR thay vì CRASH, đảm bảo cây vẫn hoạt động.
102: [Normal] - SequenceNode (Composite Node): Thực thi các node con theo thứ tự từ trái sang phải. Node này hoạt động theo nguyên tắc 'tất cả hoặc không có gì' (all-or-nothing). Cụ thể: (1) Bắt đầu với node con đầu tiên; (2) Nếu node con trả về Success, chuyển sang node con tiếp theo; (3) Nếu node con trả về Failure hoặc Error, SequenceNode dừng ngay lập tức và trả về Failure; (4) Nếu tất cả node con đều trả về Success, SequenceNode trả về Success. SequenceNode được sử dụng cho các bước kiểm thử phụ thuộc nhau, ví dụ: trong module Auth, Sequence bao gồm discover_login_form -> test_blank_password -> test_sqli_bypass. Nếu không tìm thấy form login, hai bước sau không thể thực thi.
103: [Normal] - SelectorNode (Composite Node): Thực thi các node con theo thứ tự nhưng hoạt động theo nguyên tắc 'chỉ cần một' (first-success). Cụ thể: (1) Bắt đầu với node con đầu tiên; (2) Nếu node con trả về Success, SelectorNode dừng lại và trả về Success; (3) Nếu node con trả về Failure, chuyển sang node con tiếp theo; (4) Nếu tất cả node con đều Failure, SelectorNode trả về Failure. SelectorNode được sử dụng cho fallback behavior. Ví dụ: trong module Headers, SelectorNode chứa 5 ActionNode kiểm tra 5 header khác nhau - mỗi header hoạt động độc lập, header này fail không ảnh hưởng đến header khác. Trong module Injection, SelectorNode thử SQLi, nếu không phát hiện được, chuyển sang XSS, rồi Command Injection.
104: [Normal] - ParallelNode (Composite Node): Thực thi tất cả node con đồng thời (sử dụng Python threading). ParallelNode có tham số threshold (ngưỡng) xác định số lượng node con Success tối thiểu để ParallelNode trả về Success. Cụ thể: (1) Kích hoạt tất cả node con cùng lúc trong các luồng riêng; (2) Chờ tất cả node con hoàn thành (với timeout); (3) Đếm số lượng Success và so sánh với threshold; (4) Nếu số Success >= threshold, trả về Success, ngược lại trả về Failure. ParallelNode sử dụng cơ chế ThreadPoolExecutor với max_workers và timeout configurable. Nếu một node con bị timeout hoặc exception, các node khác vẫn tiếp tục chạy (error isolation). ParallelNode lý tưởng cho các kiểm thử độc lập như kiểm tra 5 security headers cùng lúc.
105: [Heading 3] 2.2.2. Cơ chế reset và tái sử dụng cây
106: [Normal] Một tính năng quan trọng của Behavior Tree Engine là khả năng reset trạng thái cây (tree reset) cho phép tái sử dụng cây giữa các lần chạy khác nhau hoặc giữa các target khác nhau. Khi reset, tất cả các node trong cây được đưa về trạng thái ban đầu: status = IDLE, kết quả cũ được xóa (trừ metadata cấu trúc). Cơ chế reset đệ quy từ root xuống lá, đảm bảo toàn bộ cây sẵn sàng cho lần thực thi mới. Ngoài ra, mỗi node lưu log lịch sử thực thi (execution_history) dưới dạng danh sách các lần chạy với timestamp, status và kết quả, phục vụ cho việc phân tích và debug.
107: [Heading 3] 2.2.3. Ứng dụng Behavior Tree trong kiểm thử bảo mật
108: [Normal] Việc ứng dụng Behavior Tree vào kiểm thử bảo mật web mang lại nhiều lợi ích vượt trội so với kiến trúc kiểm thử tuyến tính truyền thống:
109: [Normal] (1) Mô hình hóa luồng kiểm thử phức tạp: Behavior Tree cho phép biểu diễn các kịch bản kiểm thử có nhánh rẽ, điều kiện và vòng lặp một cách trực quan. Ví dụ: 'Nếu không tìm thấy form login, bỏ qua kiểm tra authentication; nhưng nếu tìm thấy, thử blank password trước, nếu không được thì thử SQLi bypass' - đây là một SelectorTree đơn giản.
110: [Normal] (2) Dễ dàng mở rộng: Thêm một module kiểm thử mới chỉ cần xây dựng các ActionNode tương ứng và ghép chúng vào cây hiện có thông qua CompositeNode. Kiến trúc plugin cho phép thêm/bớt module mà không sửa đổi code engine.
111: [Normal] (3) Parallel execution: Các kiểm thử độc lập (ví dụ: kiểm tra 5 security headers) được thực thi song song, giảm đáng kể tổng thời gian kiểm thử. Trong hệ thống có 24 test cases, parallel execution giúp giảm thời gian từ 24 lượt request tuần tự xuống còn 5-6 lượt song song (tùy theo topology cây).
112: [Normal] (4) Tái sử dụng: Cây kiểm thử có thể được xây dựng một lần và sử dụng lại cho nhiều ứng dụng đích khác nhau (reusable test trees). Các cây con (subtree) cho từng module có thể được kết hợp linh hoạt.
113: [Normal] (5) Khả năng chịu lỗi: Khi một node gặp lỗi (network timeout, exception), lỗi được cách ly trong node đó và không ảnh hưởng đến các node khác. ParallelNode có cơ chế error isolation riêng cho từng luồng.
115: [Heading 1] CHƯƠNG 3: KIẾN TRÚC HỆ THỐNG VÀ KẾT QUẢ THỰC NGHIỆM
116: [Heading 2] 3.1. Kiến trúc tổng thể WebSec Test
117: [Normal] WebSec Test được thiết kế theo kiến trúc 3 tầng (3-tier architecture) với sự phân tách rõ ràng giữa các thành phần, đảm bảo tính module hóa, khả năng mở rộng và dễ bảo trì. Kiến trúc này bao gồm:
118: [Normal] Tầng 1 - CLI Entry Point (websec_test/main.py): Đây là điểm khởi đầu của ứng dụng, chịu trách nhiệm parsing tham số dòng lệnh bằng thư viện argparse, xác thực đầu vào (target URL format, credentials format), khởi tạo SessionClient, xây dựng Behavior Tree dựa trên module list, điều phối luồng thực thi và thu thập kết quả. Các tham số dòng lệnh hỗ trợ: --target (bắt buộc), --auth (tùy chọn, format user:pass), --modules (space-separated, mặc định chạy tất cả), --all, --output (mặc định ./reports), --timeout (mặc định 10s), --verbose.
119: [Normal] Tầng 2 - Core Engine: Bao gồm hai thành phần chính: (a) SessionClient (websec_test/client/session.py): HTTP client wrapper xây dựng trên thư viện requests, hỗ trợ quản lý session tự động (cookie persistence), CSRF token extraction (phân tích HTML response để tìm hidden input fields chứa token), headers tùy chỉnh, verify SSL và timeout configurable. (b) Behavior Tree Engine (websec_test/engine/): Thành phần cốt lõi của hệ thống, bao gồm các lớp node (ActionNode, SequenceNode, SelectorNode, ParallelNode), cơ chế tick (thực thi cây), reset state, log history và error handling.
120: [Normal] Tầng 3 - Modules và Reporting: (a) Modules (websec_test/modules/): 5 module kiểm thử độc lập được đăng ký vào hệ thống thông qua cơ chế plugin. Mỗi module có interface thống nhất gồm discover(client, target) để khám phá endpoint, và test(client, target, endpoints) để thực thi kiểm thử. (b) ResultCollector (websec_test/results/collector.py): Thu thập và tổng hợp kết quả từ tất cả module, thực hiện deduplication (loại bỏ test case trùng lặp) và aggregation. (c) Reporter (websec_test/results/reporter.py): Xuất kết quả ra hai định dạng: terminal với color-coded (PASS = xanh, FAIL = đỏ, WARN = vàng, ERROR = magenta) và JSON file với timestamp để phân tích sau.
121: [Heading 3] 3.1.1. Behavior Tree Engine - Thiết kế chi tiết
122: [Heading 3] Hình 3.1: Kiến trúc 3 tầng của WebSec Test
124: [Normal] Behavior Tree Engine là trái tim của hệ thống WebSec Test, được thiết kế với các nguyên tắc SOLID và patterns từ Domain-Driven Design. Engine được tổ chức thành một package riêng (websec_test/engine/) bao gồm các thành phần sau:
125: [Normal] - BaseNode (Abstract Base Class): Định nghĩa interface thống nhất cho tất cả node. Mỗi node có: id (string duy nhất), name (tên hiển thị), status (IDLE | RUNNING | SUCCESS | FAILURE | ERROR), children (danh sách node con), metadata (dict lưu thông tin bổ sung). Phương thức chính: tick() - thực thi node và trả về status; reset() - đưa node về trạng thái IDLE; add_child(node) - thêm node con; get_children() - lấy danh sách node con. BaseNode sử dụng Template Method pattern: tick() định nghĩa khung xử lý (pre-tick -> execute -> post-tick), các lớp con chỉ cần override _execute().
126: [Normal] - ActionNode: Kế thừa BaseNode, thực thi một hàm callback (func) được truyền vào qua constructor. ActionNode có cấu trúc: (1) pre-tick: kiểm tra nếu node đã ở trạng thái RUNNING, bỏ qua; (2) execute: gọi func, bắt exception, ghi nhận thời gian thực thi; (3) post-tick: cập nhật status dựa trên kết quả trả về, lưu log. ActionNode lưu kết quả trả về trong thuộc tính result (TestResult object).
127: [Normal] - SequenceNode: Kế thừa BaseNode, ghi đè _execute() để thực thi tuần tự các node con. Cài đặt: (1) Duyệt qua children list; (2) Gọi child.tick() cho mỗi node; (3) Nếu child trả về RUNNING, SequenceNode cũng trả về RUNNING (chờ); (4) Nếu child trả về FAILURE hoặc ERROR, dừng lại và trả về FAILURE; (5) Nếu tất cả children đều SUCCESS, trả về SUCCESS. SequenceNode duy trì con trỏ _current_index để hỗ trợ trường hợp RUNNING.
128: [Normal] - SelectorNode: Kế thừa BaseNode, ghi đè _execute() để thực thi fallback. Cài đặt: (1) Duyệt qua children list; (2) Gọi child.tick() cho mỗi node; (3) Nếu child trả về SUCCESS, dừng lại và trả về SUCCESS; (4) Nếu child trả về RUNNING, SelectorNode cũng trả về RUNNING; (5) Nếu tất cả children đều FAILURE, trả về FAILURE.
129: [Normal] - ParallelNode: Kế thừa BaseNode, ghi đè _execute() để thực thi đồng thời. Cài đặt: (1) Sử dụng concurrent.futures.ThreadPoolExecutor để chạy các node con trong các luồng riêng; (2) Tham số max_workers (mặc định = số node con) và timeout (mặc định = 30s); (3) Thu thập kết quả từ tất cả luồng; (4) Đếm số SUCCESS và so sánh với threshold; (5) Trả về SUCCESS nếu số SUCCESS >= threshold, FAILURE nếu ngược lại, ERROR nếu có exception không xử lý được. Error isolation: mỗi luồng chạy trong try/except riêng, exception chỉ ảnh hưởng đến node đó.

257: [Heading 3] 3.1.3. Các module kiểm thử - Chi tiết cài đặt
258: [Normal] Mỗi module kiểm thử trong WebSec Test tuân theo một interface thống nhất, cho phép engine gọi một cách đa hình (polymorphism). Interface bao gồm hai phương thức:
259: [Normal] discover(client: SessionClient, target: str) -> List[Endpoint]: Phương thức này khám phá các endpoint liên quan đến module. Ví dụ, Auth module discover() gửi GET request đến target, phân tích HTML response để tìm form có input type=password, trả về danh sách Endpoint (url, method) của các form login tìm được. Headers module discover() trả về endpoint mặc định (target root) vì security headers thường được áp dụng global.
260: [Normal] test(client: SessionClient, target: str, endpoints: List[Endpoint]) -> List[TestResult]: Phương thức này thực thi các kiểm thử trên các endpoint đã khám phá. Kết quả trả về dưới dạng danh sách TestResult objects.
261: [Normal] Chi tiết từng module:
262: [Normal] - Headers Module: Trong test(), gửi GET request đến từng endpoint, kiểm tra response.headers với 5 header checks được định nghĩa trong HEADER_CHECKS dictionary. Mỗi check có: test_name, severity (HIGH/MEDIUM/LOW), check_function (hàm lambda kiểm tra header), fix_recommendation. 5 checks tương ứng với 5 ActionNode trong một SelectorNode.
263: [Normal] - Auth Module: discover() trả về danh sách form login (nếu có). test() thực hiện 3 bước tuần tự trong SequenceNode: (1) test_blank_password - gửi POST request với username='admin', password=''; (2) test_sqli_bypass - gửi username với payloads như "' OR '1'='1, "admin' --; (3) test_default_credentials - thử các cặp username/password mặc định. Kết quả mỗi bước là một TestResult.
264: [Normal] - CSRF Module: discover() trả về danh sách form POST. test() thực hiện: (1) Phân tích HTML của form để tìm CSRF token (hidden input field có name chứa 'csrf', 'token', 'authenticity_token' hoặc '_token'); (2) Nếu không tìm thấy token -> test FAIL; (3) Nếu tìm thấy token -> ghi nhận PASS, sau đó thử gửi form với token cũ (reuse test) -> kiểm tra máy chủ có chấp nhận token cũ không.
265: [Normal] - Injection Module: discover() trả về danh sách form inputs (bất kỳ input field nào). test() thực thi 3 ActionNode song song: (1) SQLi test - gửi từng payload trong SQLI_PAYLOADS (15+ payloads) qua form inputs; (2) XSS test - gửi payloads trong XSS_PAYLOADS (10+ payloads) chứa thẻ script và event handler; (3) CMD test - gửi payloads trong CMD_INJECT_PAYLOADS (10+ payloads). Mỗi response được kiểm tra với dấu hiệu đặc thù: SQL syntax error messages, payload xuất hiện trong response (XSS), output lệnh hệ thống (Command).
266: [Normal] - Authz Module: discover() trả về danh sách các đường dẫn nhạy cảm từ COMMON_PATHS (30+ paths). test() thực thi song song hai kiểm tra: (1) Forced Browsing - gửi GET đến từng path trong COMMON_PATHS, kiểm tra response status (nếu 200 -> FAIL); (2) IDOR - gửi GET đến /user/1, /user/2, /user/3 (nếu tồn tại), so sánh response content (nếu khác nhau và đều 200 -> FAIL).
267: [Heading 2] 3.2. Luồng hoạt động của hệ thống
268: [Normal] Khi người dùng chạy lệnh python -m websec_test.main --target <URL> --all, hệ thống thực hiện theo luồng sau:
269: [Normal] Bước 1 - Khởi tạo: argparse parsing tham số dòng lệnh, kiểm tra tính hợp lệ của target URL, khởi tạo SessionClient với timeout và auth từ tham số --auth (nếu có). Xác định danh sách module cần chạy (từ --modules hoặc --all = tất cả 5 module).
270: [Normal] Bước 2 - Xây dựng Behavior Tree: Engine xây dựng cây test case dựa trên danh sách module. Cấu trúc cây: Root (SequenceNode) -> Child cho mỗi module (ParallelNode) -> Mỗi module -> Module-specific tree (Sequence/Selector/Parallel). Root là SequenceNode đảm bảo các module chạy tuần tự (dễ debug, tránh xung đột tài nguyên). Bên trong mỗi module, cây con có thể sử dụng Parallel (headers, authz), Sequence (auth) hoặc Selector (injection, csrf) tùy theo nhu cầu.
271: [Normal] Bước 3 - Thực thi cây (Tree Tick): Bắt đầu từ root node, engine gọi tick() đệ quy. Root SequenceNode gọi tick() cho từng module child. Mỗi module child (ParallelNode) kích hoạt các ActionNode tương ứng. Engine ghi nhận thời gian bắt đầu và kết thúc của toàn bộ quá trình.
272: [Normal] Bước 4 - Thu thập và báo cáo kết quả: Sau khi root node trả về kết quả, engine thu thập tất cả TestResult từ các ActionNode thông qua duyệt cây (BFS/DFS). ResultCollector thực hiện deduplication (nếu có test case trùng lặp giữa các module). Reporter xuất kết quả ra terminal với color-coding và lưu JSON report ra file với timestamp.

372: [Heading 2] 4.1. Giải pháp sửa lỗi và phòng ngừa cho từng lớp lỗ hổng
373: [Normal] Dựa trên kết quả kiểm thử từ WebSec Test, các giải pháp khắc phục và phòng ngừa được đề xuất cho từng lớp lỗ hổng như sau:
374: [Normal] - Đối với Security Headers: (1) Bổ sung tất cả header bảo mật vào cấu hình web server (Apache, Nginx, IIS) hoặc application framework (Django, Flask, Express). (2) Sử dụng các thư viện chuyên dụng như helmet.js (Node.js), django-csp (Python), secure-headers (Java). (3) Thiết lập HSTS với max-age >= 1 nam và includeSubDomains. (4) CSP nên được cấu hình chi tiết, tránh sử dụng unsafe-inline hoặc wildcard (*). (5) Kiểm tra định kỳ bằng các công cụ như Mozilla Observatory, securityheaders.com.
375: [Normal] - Đối với Authentication: (1) Luôn sử dụng HTTPS cho toàn bộ ứng dụng, đặc biệt là form login. (2) Không chấp nhận mật khẩu trống - validate tất cả input trước khi xử lý. (3) Sử dụng parameterized queries hoặc ORM để tránh SQL Injection trong truy vấn xác thực. (4) Implement rate-limiting (vd: 5 lần thất bại trong 15 phút -> khóa 30 phút). (5) Sử dụng multi-factor authentication (MFA) cho tài khoản quản trị. (6) Hash mật khẩu với thuật toán mạnh (bcrypt, Argon2) trước khi lưu.
376: [Normal] - Đối với CSRF: (1) Implement CSRF token cho tất cả form POST/PUT/DELETE. (2) Token nên được tạo ngẫu nhiên, gắn với session, có thời hạn (expiry). (3) Sử dụng SameSite cookie attribute (Strict hoặc Lax) cho session cookie. (4) Kiểm tra Referer/Origin header cho các request nhạy cảm. (5) Framework hiện đại (Django, Laravel, Spring) có built-in CSRF protection - nên sử dụng.
377: [Normal] - Đối với Injection: (1) Luôn sử dụng parameterized queries (prepared statements) cho tất cả truy vấn SQL - đây là biện pháp hiệu quả nhất chống SQLi. (2) Escape tất cả output HTML (context-aware escaping) để chống XSS. (3) Sử dụng CSP header như lớp phòng thủ thứ hai chống XSS. (4) Tránh gọi system() hoặc subprocess với input từ người dùng; nếu bắt buộc, sử dụng allowlist nghiêm ngặt. (5) Sử dụng thư viện validation như bleach (Python) để làm sạch HTML input.
378: [Normal] - Đối với Authorization: (1) Áp dụng nguyên tắc đặc quyền tối thiểu (Principle of Least Privilege - PoLP). (2) Kiểm tra quyền trên mọi request, không chỉ dựa vào URL hiding. (3) Sử dụng Role-Based Access Control (RBAC) hoặc Attribute-Based Access Control (ABAC). (4) Không sử dụng ID số tuần tự cho tài nguyên nhạy cảm - thay bằng UUID hoặc hash. (5) Implement server-side authorization check cho mọi API endpoint (không chỉ client-side).
379: [Normal] 4.2. Khuyến nghị cụ thể cho ứng dụng Nhom_2s (Note Basement)
380: [Normal] Dựa trên kết quả kiểm thử thực tế, nhóm tác giả đưa ra các khuyến nghị bảo mật cụ thể cho ứng dụng Nhom_2s như sau:
381: [Normal] a) Bổ sung Security Headers (Mức ưu tiên: Cao)
382: [Normal] Có thể bổ sung security headers qua cấu hình Tomcat trong file web.xml hoặc qua một servlet filter. Các headers cần thiết lập:
  - Strict-Transport-Security: max-age=31536000; includeSubDomains
  - Content-Security-Policy: default-src 'self'; script-src 'self'
  - X-Frame-Options: DENY
  - X-Content-Type-Options: nosniff
  - Referrer-Policy: strict-origin-when-cross-origin
  - Permissions-Policy: geolocation=(), microphone=(), camera=()
  - Cross-Origin-Opener-Policy: same-origin
  - Cross-Origin-Resource-Policy: same-origin

Với Tomcat, có thể thêm filter HeaderFilter trong web.xml hoặc sử dụng HttpServletResponse.setHeader() trong AuthFilter.java.
383: [Normal] b) Triển khai Rate Limiting trên Login (Mức ưu tiên: Trung bình)
384: [Normal] Cần giới hạn số lần đăng nhập thất bại trong một khoảng thời gian. Giải pháp đơn giản: lưu số lần thất bại theo IP trong bộ nhớ (HashMap<IP, Counter>) và trả về HTTP 429 After N lần thất bại liên tiếp. Có thể triển khai trực tiếp trong LoginController.java trước khi gọi userService.login().

Mã giả:
  - Kiểm tra IP trong failAttempts map
  - Nếu vượt quá 5 lần trong 15 phút → trả về 429 Too Many Requests
  - Nếu đăng nhập thành công → xóa counter cho IP đó
385: [Normal] c) Kiểm tra mật khẩu rỗng (Mức ưu tiên: Thấp)
386: [Normal] Bổ sung kiểm tra độ dài mật khẩu ở tầng backend (LoginController.java): nếu password rỗng hoặc độ dài nhỏ hơn 6 ký tự, trả về lỗi ngay lập tức thay vì gọi UserService.login().

Các biện pháp bổ sung: ràng buộc mật khẩu ít nhất 8 ký tự, bao gồm chữ hoa, chữ thường, số và ký tự đặc biệt.
387: [Normal] d) Lưu ý về kết quả dương tính giả (Forced Browsing)
388: [Normal] 14 đường dẫn forced browsing báo FAIL thực chất là dương tính giả. Tất cả đều trả về cùng một trang lỗi của Tomcat với nội dung giống nhau (content length 4122), không phải các endpoint thực sự tồn tại. Tuy nhiên, để an toàn, khuyến nghị cấu hình Tomcat trả về mã trạng thái 404 thay vì 200 cho các trang không tồn tại, bằng cách thêm <error-page> vào web.xml.
389: [Heading 2] 4.2. Giải pháp phát triển module kiểm thử mới
390: [Normal] WebSec Test được thiết kế với kiến trúc plugin cho phép cộng đồng dễ dàng đóng góp module kiểm thử mới. Quy trình phát triển module mới gồm 3 bước:
391: [Normal] Bước 1 - Tạo file module: Tạo file Python mới trong thư mục websec_test/modules/, ví dụ: websocket.py. Import các thư viện cần thiết từ core engine: Endpoint (namedtuple), TestResult, TestStatus, Severity, SessionClient.
392: [Normal] Bước 2 - Implement class module: Tạo class với hai phương thức bắt buộc: (a) discover(client, target) -> List[Endpoint]: Khám phá các endpoint liên quan bằng cách gửi request, crawl HTML, hoặc đọc cấu hình. (b) test(client, target, endpoints) -> List[TestResult]: Thực thi kiểm thử trên các endpoint. Sử dụng các node Behavior Tree nếu cần logic phức tạp (ví dụ: Sequence cho các bước phụ thuộc, Parallel cho kiểm thử song song).
393: [Normal] Bước 3 - Đăng ký module: Thêm module vào module map trong websec_test/main.py: thêm tên module vào ALL_MODULES list và thêm entry vào module_map dictionary. Module sẽ tự động được phát hiện và tích hợp vào Behavior Tree khi chạy với tham số --all hoặc --modules <tên_module>.
394: [Normal] Ví dụ module mẫu: Một module kiểm tra WebSocket Security có class WebSocketModule với discover() dò tìm endpoint WebSocket (ws://, wss://) và test() kiểm tra xác thực, mã hóa, rate-limiting trên WebSocket connection.
395: [Heading 2] 4.3. Giải pháp mở rộng Behavior Tree Engine
396: [Normal] Behavior Tree Engine hiện tại có thể được mở rộng theo nhiều hướng để đáp ứng các nhu cầu kiểm thử phức tạp hơn:
397: [Normal] - DecoratorNode: Một loại node mới bao bọc (wrap) một node con và thay đổi hành vi của nó. Các loại Decorator: (1) Inverter - đảo ngược kết quả (Success -> Failure, Failure -> Success); (2) Repeater - lặp lại node con N lần hoặc đến khi thành công; (3) Timeout - hủy node con nếu chạy quá thời gian cho phép; (4) Retry - thử lại node con N lần nếu thất bại; (5) ForceFailure - luôn trả về Failure bất kể kết quả node con (dùng cho testing).
398: [Normal] - ConditionNode: Node kiểm tra điều kiện trước khi thực thi các node con. ConditionNode kiểm tra một predicate (hàm trả về boolean), nếu True thì cho phép thực thi subtree, nếu False thì trả về Failure. Hữu ích cho việc kiểm tra: target có online không, có form login không, phiên bản Python có phù hợp không.
399: [Normal] - SubTreeNode: Node tham chiếu đến một cây con (subtree) đã được định nghĩa trước, cho phép tái sử dụng cấu trúc cây phức tạp. SubTreeNode là nền tảng cho việc xây dựng thư viện test case patterns dùng chung.
400: [Normal] - Blackboard: Cơ chế chia sẻ dữ liệu giữa các node trong cây thông qua một bảng dữ liệu chung (shared data store). Blackboard cho phép các node ghi và đọc dữ liệu (ví dụ: module Auth ghi session cookie lên Blackboard, module Authz đọc session cookie đó để kiểm tra phân quyền sau khi đã xác thực).
401: [Heading 2] 4.4. Tích hợp CI/CD và tự động hóa quy trình kiểm thử
402: [Normal] WebSec Test được thiết kế để tích hợp dễ dàng vào quy trình CI/CD (Continuous Integration/Continuous Deployment) hiện có:
403: [Normal] - Exit code convention: Hệ thống trả về exit code 0 nếu tất cả test pass (không có FAIL hoặc ERROR), exit code 1 nếu có bất kỳ FAIL hoặc ERROR nào. Điều này tương thích với chuẩn của hầu hết CI/CD pipeline (GitHub Actions, GitLab CI, Jenkins).
404: [Normal] - JSON output: Kết quả kiểm thử được lưu dưới dạng JSON file trong thư mục reports/ với timestamp, bao gồm: target URL, thời gian chạy, tổng thời gian, summary (total/pass/fail/warn/error), và danh sách chi tiết từng test result. JSON format có thể được phân tích bởi các tools khác (ví dụ: Elasticsearch, Grafana) để xây dựng dashboard.
405: [Normal] - Scheduling: Có thể chạy WebSec Test định kỳ thông qua cron job (Linux) hoặc Task Scheduler (Windows) để kiểm thử bảo mật tự động hàng ngày hoặc hàng tuần. Kết quả có thể được gửi qua email, Slack, hoặc webhook khi phát hiện lỗ hổng mới.
406: [Normal] - Docker support: Hệ thống có thể được đóng gói thành Docker image để chạy nhất quán trên mọi môi trường, tránh vấn đề dependency conflict. Docker image có thể được tích hợp vào Kubernetes Job để chạy kiểm thử on-demand.
407: [Heading 2] 4.5. Giải pháp giám sát và phát hiện (Detection Engineering)
408: [Normal] Bên cạnh kiểm thử chủ động, việc thiết lập giám sát và phát hiện tấn công tự động đóng vai trò quan trọng trong chiến lược bảo mật tổng thể. Đối với ứng dụng web, các giải pháp giám sát cần tập trung vào phân tích log truy cập, phát hiện bất thường trong traffic và cảnh báo sớm các dấu hiệu tấn công:
409: [Normal] - Web Application Firewall (WAF): Triển khai WAF (ModSecurity, AWS WAF, Cloudflare WAF, Azure Application Gateway) để phát hiện và chặn các tấn công Layer 7 như SQL Injection, XSS, Command Injection. WAF sử dụng bộ luật OWASP Core Rule Set (CRS) với hàng trăm luật phát hiện pattern tấn công phổ biến. Cảnh báo nên được kích hoạt khi một IP gửi nhiều request chứa payload độc hại trong thời gian ngắn (ví dụ: > 10 request/phút với SQLi payload).
410: [Normal] - Phân tích Web Server Logs: Thu thập và phân tích access logs (Apache, Nginx, IIS) để phát hiện các dấu hiệu tấn công: (a) SQLi — request chứa ký tự đặc biệt như ', ", OR, UNION, SELECT trong tham số GET/POST; (b) XSS — request chứa thẻ script, alert(), onerror, onload trong URL hoặc form data; (c) Path Traversal — request chứa ../, %2e%2e/, ....// trong đường dẫn; (d) Rate-limiting violation — một IP gửi > 100 request/phút. Các công cụ như ELK Stack (Elasticsearch, Logstash, Kibana), Splunk, hay Graylog có thể được sử dụng để tổng hợp log, xây dựng dashboard và luật phát hiện real-time.
411: [Normal] - Giám sát xác thực và Session: (a) Phát hiện brute-force login — theo dõi số lần đăng nhập thất bại từ một IP (ngưỡng: 5 lần / 15 phút); (b) Phát hiện CSRF token reuse — token được sử dụng nhiều hơn một lần cho request POST; (c) Phát hiện session hijacking — cùng session ID được dùng từ hai địa chỉ IP khác nhau trong khoảng thời gian ngắn; (d) Phát hiện hành vi bất thường — user thực hiện request với User-Agent lạ hoặc ngoài giờ làm việc. Các luật phát hiện này có thể được triển khai trên SIEM (Splunk, Wazuh, ELK) hoặc tích hợp vào hệ thống OWASP AppSensor.

Thứ nhất, toàn bộ 8 security headers đều chưa được thiết lập, bao gồm HSTS, CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, Cross-Origin-Opener-Policy và Cross-Origin-Resource-Policy. Điều này khiến ứng dụng dễ bị tấn công clickjacking, XSS dạng cũ, và thiếu cơ chế bảo vệ vận chuyển. Bên cạnh đó, hệ thống gửi password rỗng cũng được chấp nhận ở tầng xử lý (cảnh báo mức WARN). Các module cors, cookies, disclosure và methods đều đạt PASS, cho thấy cấu hình CORS an toàn, không lộ thông tin máy chủ và không cho phép các HTTP method nguy hiểm.
